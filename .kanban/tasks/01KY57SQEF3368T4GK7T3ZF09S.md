## What

Turn `get lines` polling from busy-poll (many tool calls, each burning transcript context) into long-poll, and tell the model whether to keep polling.

Files:
- `Sources/ShellTool/Operations/GetLines.swift`:
  - `waitSeconds: Int?`, no `@OperationParam` short flag (pinned to match `execute command`). Negative `waitSeconds` returns the corrective message `"waitSeconds must be non-negative"` (identical to ExecuteCommand's pin) via `GetLinesOutput.corrective`.
  - Semantics: if lines already satisfy the request, return immediately. Otherwise, while the command's record is `running` and the deadline has not elapsed, re-query on a short cadence (~200ms) until lines appear in the requested range, the command leaves `running`, or the deadline fires. The poll loop lives in `GetLines.execute(in:)` — NOT on `ShellState` (its pinned contract is "state is brief; all methods O(small)"). Unknown `commandID` returns immediately (empty, parity). Cancellation of the awaiting task ends the wait like the deadline does: the current snapshot returns promptly, never a hot zero-delay re-read loop.
  - `LineRange` gains `status: String?` — the command's current status raw value (`running`/`completed`/`killed`/`timed_out`), omitted for an unknown id (synthesized optional encoding). This is how the model knows a poll loop should end.

## Acceptance Criteria
- [x] Lines already stored in range: returns immediately even with `waitSeconds` set
- [x] Long-poll returns early (well before the deadline) once a running command emits a line in the requested range

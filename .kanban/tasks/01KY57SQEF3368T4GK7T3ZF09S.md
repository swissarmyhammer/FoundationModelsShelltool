## What

Turn `get lines` polling from busy-poll (many tool calls, each burning transcript context) into long-poll, and tell the model whether to keep polling.

Files:
- `Sources/ShellTool/Operations/GetLines.swift`:
  - Add `waitSeconds: Int?` with `@Guide` text like: "Seconds to wait for lines to appear when the requested range is empty and the command is still running (optional, default: return immediately)". **No `@OperationParam` short flag** — pinned to match `execute command`. **Negative `waitSeconds` returns the corrective message `"waitSeconds must be non-negative"`** (identical to the ExecuteCommand task's pin; this op returns it via its output the way `ExecuteOutput.corrective` does — add a corrective-capable output wrapper if needed).
  - Semantics: if lines already satisfy the request, return immediately (waitSeconds irrelevant). Otherwise, while the command's record is `running` and the deadline has not elapsed, re-query on a short cadence (~200ms `Task.sleep` loop) until lines appear in the requested range, the command leaves `running`, or the deadline fires. **The poll loop lives in `GetLines.execute(in:)` — NOT on `ShellState`** (the actor's pinned contract is "state is brief, execution is long; all methods O(small)", and each iteration re-reads the log file, which belongs in the op). Unknown `commandID` returns immediately (empty, parity).
  - `LineRange` gains `status: String?` — the command's current status raw value (`running`/`completed`/`killed`/`timed_out`), omitted for an unknown id (synthesized optional encoding, same technique as `ProcessRow.exitCode`). This is how the model knows a poll loop should end.

Depends only on incremental recording (lines must be visible mid-run); tests drive `ShellRunner` in a background test task, so this proceeds in parallel with the ExecuteCommand task.

Merge hygiene: the ExecuteCommand task runs in parallel and also edits `FusionTests.swift`/`CLIConvergenceTests.swift`. Add this task's schema/CLI assertions as **new, clearly-owned test functions** (e.g. `getLinesWaitSecondsSchema...`) — never edit a shared test function both tasks touch.

## Acceptance Criteria
- [ ] Lines already stored in range: returns immediately even with `waitSeconds` set
- [ ] Long-poll returns early (well before the deadline) once a running command emits a line in the requested range
- [ ] Deadline with no new lines: returns empty `lines`, `status: "running"`
- [ ] Command completes during the poll with no lines in range: returns promptly with `status: "completed"`
- [ ] Finished command: `status` reflects its final state; unknown `commandID`: empty result, `status` key absent from JSON; negative `waitSeconds`: pinned corrective message
- [ ] Fused schema and CLI expose `waitSeconds` with no short flag
